# Лабораторные работы по дисциплине «Проектирование пользовательских интерфейсов»
## Базовый проект: веб‑система рецензирования музыкального творчества

В качестве единого проектного примера во всех лабораторных работах используется веб‑приложение для рецензирования музыкального творчества. Система позволяет пользователям:
- просматривать каталог альбомов и треков;
- оставлять многокритериальные оценки (текст, структура, стиль, индивидуальность, атмосфера/«вайб»);
- публиковать текстовые рецензии, проходящие модерацию;
- взаимодействовать с исполнителями через отзывы и специальные отметки.


## Лабораторная работа №1. Анализ качества пользовательского интерфейса

### Задание 2. Обоснование критериев качества интерфейса

**Формулировка задания**

Обосновать критерии качества проектируемого пользовательского интерфейса, используя восемь основных характеристик, рассмотренных в лекционном материале по дисциплине.

Для каждого из восьми критериев необходимо:

1. Чётко сформулировать саму характеристику (краткое определение одним–двумя предложениями).
2. Описать, как данный критерий проявляется в интерфейсе веб‑системы рецензирования музыкального творчества:
   - на каких экранах он особенно важен (главная страница, каталог релизов, форма добавления рецензии, панель модератора, профиль исполнителя и т.п.);
   - какие решения по дизайну и взаимодействию пользователей помогают этому критерию быть выполненным.
3. Привести не менее одного конкретного примера или сценария использования для каждого критерия:
   - что делает пользователь;
   - какие элементы интерфейса он видит;
   - почему с точки зрения данного критерия это хорошо (или какую проблему нужно учесть в дальнейшей доработке).

**Требуемый результат**

По итогам выполнения задания студент должен подготовить текстовый документ (или раздел отчёта к ЛР1), в котором:

- перечислены восемь критериев качества пользовательского интерфейса;
- для каждого критерия:
  - дано краткое определение;
  - приведено обоснование применительно к интерфейсу системы рецензирования музыки;
  - приведён пример из проектируемого интерфейса (описание экрана, сценария или макета).

Полученный материал будет использоваться далее при проектировании и оценке прототипов интерфейса в последующих лабораторных работах.



## Лабораторная работа №2. Проектирование макетов пользовательского интерфейса

### Задание 2. Макеты интерфейса веб‑системы рецензирования музыки

**Формулировка задания**

Представить набор макетов пользовательского интерфейса веб‑приложения для рецензирования музыкального творчества и описать основные функции системы с привязкой к элементам интерфейса.

В качестве ориентира можно использовать пример из файла «Макеты интерфейсов.pdf» (структура экранов, уровень детализации и стиль оформления).

**Состав макетов**

В минимальный комплект макетов должны входить:

1. **Главная страница / «Актуальное»**
   - блок с актуальными релизами (альбомы и треки);
   - блок с последними или популярными рецензиями / лайками;
   - элементы навигации по разделам: каталог, поиск, профиль пользователя, профиль исполнителя, админ‑панель (для модератора).

2. **Страница альбома / трека**
   - обложка, название, исполнитель, год, жанр;
   - агрегированные оценки по критериям (рифмы/образы, структура/ритмика, стиль, индивидуальность, атмосфера/вайб);
   - список рецензий с краткой выжимкой и возможностью перейти к полному тексту;
   - кнопки «Оценить», «Добавить рецензию», «Лайкнуть» и т.п.

3. **Форма добавления рецензии**
   - элементы ввода оценок по критериям (ползунки или выпадающие списки 1–10);
   - автоматический вывод итогового балла;
   - поле для текстовой рецензии с пометкой о том, что она будет отправлена на модерацию;
   - кнопки «Отправить» и «Отмена».

4. **Панель модератора**
   - список рецензий «на модерации» с указанием автора, альбома/трека, выставленных оценок и даты;
   - элементы управления: кнопки «Одобрить», «Отклонить», при необходимости — поле для комментария модератора;
   - фильтры/переключатели по статусу рецензий (на модерации, одобренные, отклонённые).

5. **Профиль исполнителя**
   - информация об артисте (имя, описание, жанр/стили);
   - список релизов с краткими показателями оценок;
   - блок рецензий на релизы исполнителя с возможностью отметить рецензии как «полезные» со стороны артиста.

**Требуемый результат**

Для каждого макета необходимо:

- изобразить расположение основных элементов интерфейса (допустимы наброски в Figma, Balsamiq, Draw.io или аккуратные схемы от руки);
- подписать элементы управления (кнопки, поля ввода, списки), указав, какие функции системы за ними стоят;
- кратко описать текстом, какие пользовательские сценарии поддерживает данный экран (например: «пользователь выбирает альбом и читает рецензии», «модератор просматривает очередь новых отзывов и принимает решение»).

Все макеты и описания включаются в отчёт по лабораторной работе №2 или в отдельный раздел проектной документации по системе рецензирования музыкального творчества.


## Лабораторная работа №3. Принципы хорошего дизайна и эвристики юзабилити

### Задание 1. Применение принципов Дитера Рамса и эвристик Якоба Нильсена

**Формулировка задания**

Проиллюстрировать на макетах пользовательского интерфейса веб‑системы рецензирования музыкального творчества использование принципов хорошего дизайна Дитера Рамса и эвристических правил Якоба Нильсена. Необходимо рассмотреть не менее **семи** характеристик (в сумме по двум авторам) и показать, как они реализованы в проектируемом интерфейсе.

В качестве справочного материала используются раздаточные файлы:
- «10 правил Дитера Рамса»;
- «Эвристические правила Якоба Нильсена».

**Что нужно сделать**

1. Выбрать из перечней Рамса и Нильсена как минимум 7 принципов/эвристик. Рекомендуется включить:
   - 3–4 принципа Рамса (например, «хороший дизайн понятен», «ненавязчив», «честен», «минималистичен»);
   - 3–4 эвристики Нильсена (например, «видимость состояния системы», «соответствие реальному миру», «свобода действий и отмены», «последовательность и стандарты»).

2. Для каждого выбранного принципа/эвристики:
   - кратко сформулировать его содержание своими словами;
   - указать **конкретный экран** или элемент интерфейса твоей системы рецензий, на котором этот принцип реализуется  
     (главная страница, страница альбома, форма добавления рецензии, панель модератора, профиль исполнителя и т.п.);
   - объяснить, какие именно решения дизайна или логики работы интерфейса соответствуют этому принципу  
     (расположение элементов, подписи, обратная связь, сообщения об ошибках, минимизация шума, возможность отмены действия и т.д.).

3. Для наглядности:
   - подготовить по крайней мере один фрагмент макета (или отдельный кадр в Figma) для каждого выбранного принципа/эвристики;
   - на макете визуально выделить (рамкой/подписью) элементы, демонстрирующие данный принцип;
   - снабдить макет краткой подписью в отчёте: «На рисунке Х показано, как реализуется принцип … (Рамс/Нильсен)».

**Требуемый результат**

Отчёт по ЛР3 должен содержать:

- перечень выбранных принципов Дитера Рамса и эвристик Якоба Нильсена (не менее 7 пунктов);
- для каждого пункта:
  - краткое описание принципа;
  - ссылку на соответствующий макет интерфейса системы рецензирования музыки;
  - текстовое пояснение, как данный принцип реализован в выбранном фрагменте интерфейса;
- набор иллюстраций (скриншоты/кадры макетов) с подписями.

Материалы ЛР3 могут опираться на макеты, разработанные в лабораторной работе №2, при необходимости допускается их уточнение или дополнение.


## Лабораторная работа №4. Организация сценария работы агента‑помощника

### Задание. Разработка сценариев действий для агента‑помощника на сайте

**Формулировка задания**

1. Разработать сценарий действий для агента‑помощника в веб‑системе рецензирования музыкального творчества.
2. Описать не менее **семи различных (не однотипных) ситуаций**, в которых используются системы «помощи‑подсказки‑предупреждения‑информирования» при работе с пользовательским интерфейсом сайта.
3. Оформить результаты в виде отчёта с заполненной таблицей (структура таблицы приведена в файле к лабораторной работе).

**Требования к содержанию**

Агент‑помощник рассматривается как встроенная подсистема поддержки пользователя (всплывающие подсказки, контекстная справка, уведомления, пошаговые гиды и т.п.), работающая поверх основных экранов:

- главная страница и каталог релизов;
- страница альбома/трека и список рецензий;
- форма добавления рецензии;
- панель модератора;
- профиль исполнителя и профиль обычного пользователя.

В отчёте необходимо:

1. **Кратко описать роль агента‑помощника** в системе:
   - какие типы пользователей он поддерживает (новый пользователь, активный рецензент, модератор, исполнитель);
   - какие цели помогает достигать (разобраться в интерфейсе, корректно заполнить форму рецензии, избежать ошибок, понять статус действий и т.д.).

2. **Привести не менее семи ситуаций использования помощи**, например:
   - первое посещение сайта: краткий тур по основным разделам;
   - попытка отправить рецензию без оценок по всем критериям;
   - слишком короткий/неинформативный текст рецензии (мягкое предупреждение и подсказки, что можно улучшить);
   - переход в панель модератора впервые (объяснение статусов рецензий и доступных действий);
   - просмотр профиля исполнителя (подсказка о значении отметки «отмечено исполнителем» у рецензий);
   - поиск по каталогу, который не дал результатов (информационное сообщение и советы по уточнению запроса);
   - попытка удалить свою рецензию или изменить уже опубликованную (подтверждение действия, разъяснение последствий);
   - получение уведомления о том, что рецензия прошла модерацию или была отклонена (информационное сообщение с кратким объяснением).

3. **Заполнить таблицу сценариев** (одна строка — одна ситуация), указав для каждой:
   - название ситуации;
   - тип пользователя;
   - контекст (на каком экране и в какой момент возникает);
   - действия пользователя до появления помощи;
   - реакцию агента‑помощника (формат подсказки/предупреждения/информирования);
   - ожидаемый результат (что пользователь поймёт или сделает по‑другому).

**Требуемый результат**

Отчёт по ЛР4 должен содержать:

- общее текстовое описание концепции агента‑помощника для системы рецензирования музыки;
- заполненную таблицу с не менее чем семью различными сценариями помощи;
- при наличии макетов — ссылки на соответствующие экраны, где эти сценарии будут визуально реализованы.


## Лабораторная работа №5. Квантификация пользовательского интерфейса

### Цель работы

Через расчёт времени, требуемого для доступа к различным объектам пользовательского интерфейса веб‑системы рецензирования музыкального творчества, научиться количественно оценивать параметры диалога с пользователем (скорость, удобство, избыточность действий) на основе модели GOMS и законов Фитса и Хика [file:315].

### Задание

Необходимо выполнить лабораторную работу и представить отчёт в виде файла, включив всю информацию согласно примеру «Лабораторная работа КПИ». В отчёте использовать **реальные формы** разработанного проекта (страница альбома, форма добавления рецензии, панель модератора и т.п.), представив их в виде скриншотов.

#### Задание 1. Оценка пяти шагов интерфейса (модель GOMS)

1. Выбрать не менее **пяти реализованных шагов** взаимодействия пользователя с интерфейсом системы рецензий. В качестве шагов можно брать, например:
   - поиск и открытие страницы альбома;
   - переход к форме добавления рецензии;
   - заполнение и отправка формы многокритериальной оценки;
   - одобрение рецензии модератором;
   - отметка рецензии исполнителем как «полезной».

2. Для каждого шага:
   - описать исходную форму/экран (со скриншотом);
   - разложить действие пользователя на последовательность операторов модели GOMS (M, H, P, K и др.), по аналогии с приведённым примером в файле‑шаблоне [file:315];
   - с использованием табличных значений временных интервалов для операторов рассчитать:
     - время выполнения шага при полном вводе данных;
     - при использовании значений по умолчанию (если применимо);
     - среднее время выполнения шага с учётом вероятностей тех или иных действий пользователя.

3. Сгруппировать результаты в таблицу (аналог таблиц 1–2 в примере), сделать вывод о рациональности выбранного стиля интерфейса и количестве шагов.

#### Задание 2. Построение моделей по законам Фитса и Хика

1. **Закон Фитса.**
   - Определить набор действий, связанных с наведением курсора и кликом по элементам интерфейса (кнопка «Добавить рецензию», элементы меню, кнопки модерации и т.д.).
   - Для каждого действия выполнить не менее **30 измерений** времени перемещения/клика (желательно с использованием таймера в приложении).
   - Зафиксировать расстояние и размеры цели для каждого элемента, сформировать выборку данных и определить эмпирические коэффициенты закона Фитса для интерфейса системы.
   - Сделать вывод о том, какие элементы стоит укрупнить или перенести, чтобы сократить время доступа.

2. **Закон Хика.**
   - Выбрать интерфейсные ситуации с выбором из нескольких вариантов (например, фильтры по жанру, сортировка рецензий, выбор действий модератора).
   - Собрать не менее **30 измерений** времени принятия решения при разном количестве вариантов, учитывая, что выбор объектов **не является равновероятным** [file:315].
   - По собранным данным определить эмпирические коэффициенты закона Хика и оценить, как увеличение количества пунктов меню/фильтров влияет на время выбора в твоём интерфейсе.

3. Проверить статистическую надёжность полученных выражений (коэффициенты регрессии, качество аппроксимации) и кратко интерпретировать результаты с точки зрения улучшения интерфейса.

**Требуемый результат**

Отчёт по ЛР5 должен содержать:

- краткое описание выбранных форм и скриншоты интерфейсов системы рецензирования музыки;
- для задания 1 — таблицы с последовательностями операторов GOMS и рассчитанными временами выполнения пяти шагов;
- для задания 2 — описания экспериментов, выборки измерений (для законов Фитса и Хика), найденные коэффициенты моделей и выводы о качестве и скорости работы пользовательского интерфейса.
