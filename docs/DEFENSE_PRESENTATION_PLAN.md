# План защиты и сценарий демонстрации Mustreview

Документ нужен как личная шпаргалка перед защитой: в каком порядке показывать сайт, что говорить во время переходов и какие технологии упоминать, если комиссия задаст уточняющие вопросы.

## 1. Общая логика выступления

Рекомендуемый темп: 7-10 минут на основную демонстрацию и 2-3 минуты запасом на вопросы.

Главная идея защиты: показать не просто набор страниц, а законченный пользовательский сценарий музыкального сервиса:

1. Что это за продукт и какую проблему решает.
2. Как пользователь просматривает рецензии и находит музыку.
3. Как устроена оценка альбомов и треков.
4. Чем полезен паспорт релиза.
5. Как работает профиль, лайки, подписки и элементы геймификации.
6. Как администратор модерирует контент.
7. На чем это реализовано технически и как развернуто.

## 2. Короткое вступление

Пример речи:

> Тема моей работы - веб-сервис для музыкальных рецензий Mustreview. Идея проекта в том, чтобы объединить каталог альбомов и треков, пользовательские рецензии, систему оценок по нескольким критериям и социальные механики: лайки, подписки, профили, уровни и достижения. В отличие от простой оценки "понравилось/не понравилось", пользователь может разобрать релиз по нескольким параметрам, а сервис затем агрегирует эти оценки и показывает более наглядную картину.

Что можно добавить, если нужно:

- Проект учебный, но сделан как полноценное веб-приложение.
- Есть разделение на frontend, backend и базу данных.
- Есть админская модерация, чтобы публичная лента не наполнялась непроверенным контентом.
- Приложение развернуто на VPS через Docker.

Технологии, которые можно назвать уже здесь:

- Frontend: React 18, React Router, Axios, CSS-переменные.
- Backend: Go, Gin, GORM.
- База данных: PostgreSQL.
- Инфраструктура: Docker Compose, Nginx во frontend-контейнере, Caddy на VPS для проксирования и HTTPS.

## 3. Сценарий демонстрации сайта

### Шаг 1. Открыть сайт и показать ленту

Страница: `/feed`

Что показать:

- свежие одобренные рецензии;
- карточки с автором, релизом, оценкой, лайками и коротким текстом;
- переключатель "Все / Подписки";
- боковую сводку ленты: количество рецензий, средний балл, быстрые переходы, активные авторы.

Пример речи:

> Начинаю с ленты. Это основной экран сервиса, куда попадают только одобренные рецензии. Пользователь сразу видит автора, релиз, итоговую оценку и краткий фрагмент текста. Здесь же есть лайки и режим "Подписки", чтобы можно было смотреть не всю ленту, а только авторов, на которых пользователь подписан.

Техническая пометка:

- Данные ленты приходят с backend API через Axios.
- В ленту попадают только рецензии со статусом `approved`.
- Карточки адаптированы под мобильный экран, а шапка на мобильных превращается в раскрывающееся меню.

Переход:

> Из ленты можно перейти к конкретному альбому или треку. Покажу карточку релиза, потому что там лучше всего видно систему оценки.

### Шаг 2. Открыть страницу альбома

Страница: например `/albums/6`

Что показать:

- обложку, название, артиста, жанр;
- среднюю оценку;
- список треков;
- рецензии на альбом;
- кнопку лайка;
- иконку паспорта релиза рядом с действиями;
- блок похожих релизов внизу страницы.

Пример речи:

> На странице альбома собрана вся информация по релизу: базовые данные, треки, средняя оценка, пользовательские рецензии и похожие релизы. Средняя оценка не вводится вручную, она считается на основе одобренных рецензий. Это важно, потому что отклоненные или ожидающие модерации материалы не должны влиять на публичный рейтинг.

Техническая пометка:

- Страница получает данные по маршрутам `/api/albums/:id`, `/api/albums/:id/tracks`, `/api/reviews`.
- Похожие релизы подбираются по жанру.
- Для лайков есть отдельные таблицы и уникальные индексы, чтобы один пользователь не мог лайкнуть один и тот же объект несколько раз.

Переход:

> Но обычная средняя оценка не всегда объясняет, за счет чего релиз получил такой балл. Поэтому я добавил отдельный паспорт релиза.

### Шаг 3. Открыть паспорт релиза

Компонент: `ReleasePassport`

Что показать:

- большую итоговую оценку;
- количество рецензий полным текстом;
- чип согласия критиков;
- "ДНК звучания" в виде радара;
- гистограмму разброса мнений;
- блок "Оценки";
- блок "Итог по критериям": какие критерии лидируют и какие ниже остальных.

Пример речи:

> Паспорт релиза - это аналитическая карточка поверх страницы. Здесь итоговая оценка раскрывается визуально. Радар показывает средние значения по пяти критериям: рифмы и образы, структура и ритмика, реализация стиля, индивидуальность и атмосфера. Гистограмма показывает, насколько оценки пользователей сходятся или расходятся. Если несколько критериев имеют одинаковые максимальные или минимальные значения, они выводятся вместе, чтобы интерфейс не создавал ложное ощущение, что лидирует только один параметр.

Что важно подчеркнуть:

- Это не отдельная ручная сущность, а агрегированное представление по существующим рецензиям.
- Пользователь видит не только "79 из 90", но и структуру этой оценки.
- Цветовой чип помогает быстро понять согласованность мнений.

Техническая пометка:

- На frontend это отдельный React-компонент.
- Графики построены средствами SVG/CSS без отдельной тяжелой графической библиотеки.
- Данные берутся из массива рецензий и агрегируются на клиенте для отображения.
- Базовые оценки хранятся в PostgreSQL как поля рецензии, итоговый балл пересчитывается при создании/модерации.

Переход:

> Альбомы и треки в сервисе оцениваются одинаково, поэтому такая же логика работает и для отдельных треков.

### Шаг 4. Показать страницу трека

Страница: `/tracks/:id`

Что показать:

- карточку трека;
- оценку;
- рецензии;
- паспорт трека;
- похожие треки.

Пример речи:

> Помимо альбомов можно оценивать отдельные треки. Это важно для музыкального сервиса, потому что пользователь может не хотеть писать рецензию на весь альбом, но может оценить конкретную композицию. Механика та же: рецензии, критерии, средний балл, лайки и паспорт.

Техническая пометка:

- В модели рецензии есть связь либо с альбомом, либо с треком.
- Backend маршруты для альбомов и треков разделены, но интерфейс использует общие компоненты там, где логика совпадает.

Переход:

> Теперь покажу, как пользователь может искать музыку и смотреть подборки.

### Шаг 5. Поиск и топы

Страницы: `/search`, `/tops`

Что показать в поиске:

- поиск по артистам, альбомам и трекам;
- фильтрацию/результаты;
- переходы в карточки.

Что показать в топах:

- популярные релизы;
- лучшие по оценке;
- скрытые жемчужины;
- выбор артистов.

Пример речи:

> Помимо ленты, есть поиск и раздел топов. Поиск нужен для прямого сценария, когда пользователь уже знает артиста или релиз. Топы решают другую задачу: помогают открыть музыку через рейтинги и подборки. Здесь есть не только популярное, но и "скрытые жемчужины", чтобы сервис не сводился к самым очевидным релизам.

Техническая пометка:

- Поиск идет через backend endpoint `/api/search`.
- Топы строятся на агрегированных оценках, лайках и признаках релизов.
- Сортировки на backend защищены белыми списками колонок, чтобы не подставлять пользовательский ввод напрямую в SQL `ORDER BY`.

Переход:

> Теперь перейду к пользовательской части: профиль, предпочтения и социальные механики.

### Шаг 6. Профиль пользователя

Страница: `/profile`

Что показать:

- аватар, имя, био;
- уровень пользователя;
- статистику;
- вкладки профиля;
- предпочтения: артисты, альбомы, треки;
- рецензии и лайкнутые материалы;
- достижения.

Пример речи:

> Профиль нужен не только как страница с именем пользователя. Он показывает музыкальные предпочтения, активность, уровень и достижения. Пользователь может вручную выбрать любимых артистов, альбомы и треки. Если он ничего не выбрал, профиль не остается пустым: система может собрать блоки из его рецензий.

Техническая пометка:

- Авторизация реализована через подписанный bearer-token.
- Токен хранится на клиенте и передается в `Authorization`.
- Пароли пользователей хранятся не в открытом виде, а как bcrypt hash.
- Профиль получает данные через `/api/users/:id`.
- Любимые альбомы/треки/артисты хранятся как JSON-поля в модели пользователя.

Переход:

> Чтобы показать полный цикл пользовательского контента, дальше можно создать или открыть рецензию и затем перейти к модерации.

### Шаг 7. Создание рецензии

Где показывать: страница альбома или трека, форма создания рецензии.

Что показать:

- текст рецензии;
- пять критериев оценки;
- итоговую оценку;
- отправку на модерацию.

Пример речи:

> При создании рецензии пользователь не просто ставит один общий балл. Он оценивает релиз по пяти параметрам, а итоговый балл формируется из этих значений. После создания рецензия получает статус ожидания модерации, поэтому она не появляется в публичной ленте сразу.

Техническая пометка:

- Создание рецензии идет через `POST /api/reviews`.
- Статусы рецензии: `pending`, `approved`, `rejected`.
- После approve/reject backend пересчитывает агрегированные рейтинги релиза.
- На frontend есть обработка 401 без полной перезагрузки SPA.

Переход:

> Последний пользовательский шаг - посмотреть, как администратор управляет этим контентом.

### Шаг 8. Админка и модерация

Страница: `/admin`

Что показать:

- список pending/approved/rejected;
- счетчики;
- approve/reject;
- что после одобрения рецензия попадает в ленту.

Пример речи:

> Админская панель нужна для контроля качества публичной ленты. Администратор видит рецензии по статусам и может одобрить или отклонить материал. Это защищает сервис от случайного или неподходящего контента и одновременно делает рейтинг стабильнее, потому что публичные агрегаты считаются по одобренным рецензиям.

Техническая пометка:

- Доступ к админке защищен ролью `is_admin`.
- На backend используется middleware для проверки администратора.
- Маршруты модерации: `/api/reviews/:id/approve` и `/api/reviews/:id/reject`.
- В публичные выборки попадают только одобренные рецензии.

Переход:

> В завершение коротко покажу, как проект устроен технически и как он развернут.

## 4. Техническое завершение

Что сказать:

> Архитектурно приложение разделено на три основных слоя. Frontend написан на React, backend - на Go с Gin, база данных - PostgreSQL. Для работы с базой используется GORM, для локального запуска и деплоя - Docker Compose. На VPS приложение запущено в контейнерах: отдельно база, backend и frontend. Frontend-контейнер использует Nginx и проксирует `/api` на backend. Перед ним стоит Caddy, который принимает внешний трафик и после активации DNS будет автоматически выпускать HTTPS-сертификат.

Короткая схема:

```text
Пользователь
  -> домен / IP сервера
  -> Caddy
  -> frontend nginx
  -> React SPA
  -> /api -> Go backend
  -> PostgreSQL
```

Что упомянуть про CI/CD:

> В GitHub Actions настроены проверки backend и frontend: `go vet`, `go test`, сборка backend, сборка frontend, smoke-test production compose и публикация Docker-образов в GitHub Container Registry. На сервере используются уже готовые образы, поэтому деплой не требует сборки проекта прямо на VPS.

Техническая пометка:

- `compose.deploy.yml` использует образы `ghcr.io/afonk1n/vkr/backend:latest` и `ghcr.io/afonk1n/vkr/frontend:latest`.
- На сервере создан `.env` с секретами, паролем БД, настройками CORS и production-флагами.
- После первого запуска `SEED_ENABLED` и `DB_CREATE_ENABLED` переводятся в `false`.

## 5. Короткая финальная речь

> В результате получился работающий веб-сервис музыкальных рецензий: пользователь может искать музыку, читать и писать рецензии, оценивать альбомы и треки по нескольким критериям, взаимодействовать через лайки и подписки, вести профиль и видеть достижения. Администратор управляет публикацией рецензий через модерацию. Технически проект включает frontend, backend, базу данных, контейнеризацию, CI/CD и деплой на VPS. Основной акцент я сделал на том, чтобы оценка музыки была не просто числом, а понятной аналитикой через критерии, паспорт релиза и распределение мнений.

## 6. Быстрая шпаргалка по технологиям

| Место в демонстрации | Что используется | Как объяснить простыми словами |
| --- | --- | --- |
| Навигация между страницами | React Router | SPA не перезагружает всю страницу, а переключает представления внутри React |
| Запросы к API | Axios | Единый клиент для обращения frontend к backend |
| Состояние авторизации | AuthContext + bearer-token | Frontend хранит сессию и передает токен в запросах |
| Backend API | Go + Gin | Gin обрабатывает HTTP-маршруты и вызывает контроллеры |
| Работа с БД | GORM + PostgreSQL | GORM связывает Go-модели с таблицами PostgreSQL |
| Рецензии и оценки | Models + controllers | Рецензия хранит текст, критерии, итоговый балл и статус модерации |
| Паспорт релиза | React-компонент + SVG/CSS | Frontend агрегирует оценки и показывает их визуально |
| Лайки | Отдельные таблицы + уникальные индексы | Один пользователь не может повторно лайкнуть тот же объект |
| Модерация | AdminMiddleware | Backend проверяет роль администратора |
| Безопасная сортировка | Белые списки колонок | Пользовательский `sort_by` не подставляется напрямую в SQL |
| Деплой | Docker Compose | Каждый слой приложения запущен в отдельном контейнере |
| Продовый frontend | Nginx | Раздает собранный React и проксирует `/api` |
| HTTPS | Caddy + Let's Encrypt | Caddy сам получает и обновляет сертификат, когда DNS домена активен |
| CI/CD | GitHub Actions + GHCR | После push проходят проверки и собираются Docker-образы |

## 7. Возможные вопросы комиссии и короткие ответы

**Почему оценка не одна, а по пяти критериям?**

Так пользователь дает более содержательную оценку. Итоговый балл становится не просто субъективным числом, а складывается из параметров: текст, структура, стиль, индивидуальность и атмосфера.

**Почему нужна модерация?**

Чтобы в публичную ленту и агрегированные рейтинги попадали только проверенные рецензии. Это защищает качество данных и уменьшает риск мусорного контента.

**Почему Go на backend?**

Go подходит для простых и быстрых HTTP-сервисов, хорошо собирается в один бинарник и удобно запускается в Docker. Gin дает легкую маршрутизацию, middleware и обработчики.

**Почему PostgreSQL?**

Данные имеют связи: пользователи, рецензии, альбомы, треки, лайки, подписки. Реляционная база хорошо подходит для таких моделей и агрегированных запросов.

**Как защищена авторизация?**

Пароли хранятся как bcrypt hash. После входа пользователь получает подписанный bearer-token. Для защищенных маршрутов backend проверяет токен и роль пользователя.

**Что будет, если пользователь обновит страницу?**

React-приложение восстановит состояние авторизации из сохраненного токена и проверит текущего пользователя через `/auth/me`.

**Почему Docker?**

Docker фиксирует окружение приложения. Backend, frontend и PostgreSQL запускаются одинаково локально и на сервере, меньше ручных зависимостей.

**Что такое Caddy и зачем он нужен?**

Caddy стоит перед приложением как reverse proxy. Он принимает внешний HTTP/HTTPS-трафик, прокидывает его на frontend-контейнер и автоматически выпускает сертификат Let's Encrypt.

**Что такое punycode у домена?**

Это техническая ASCII-форма кириллического домена. `мьюзик-рейтинг.рф` внутри DNS может записываться как `xn----ftbfgfcgfpr3a0a6j2a.xn--p1ai`. Для пользователя это один и тот же адрес.

**Почему после первого запуска сидер выключается?**

Сидер нужен для начального демо-наполнения. После первого запуска его лучше выключить, чтобы production-среда не пыталась повторно создавать демо-данные при перезапусках.

## 8. Маршрут кликов во время защиты

1. Открыть `/feed`.
2. Показать ленту, карточки, лайки, переключатель "Все / Подписки".
3. Перейти в альбом из карточки.
4. Показать страницу альбома, треки, рецензии, похожие релизы.
5. Открыть паспорт релиза.
6. Пояснить радар, гистограмму, критерии и согласие критиков.
7. Перейти на страницу трека.

## 9. Что лучше не делать во время защиты

- Не углубляться в формулу итогового балла, если не спросят. Лучше объяснять критерии и идею агрегирования.
- Не показывать слишком много похожих страниц подряд. Альбом и трек достаточно показать как два примера одной механики.
- Не тратить время на регистрацию нового пользователя, если интернет или ввод данных тормозит. Лучше использовать демо-аккаунт.
- Не открывать серверные конфиги с секретами. Про `.env` можно рассказать словами.
- Не зависеть от домена, если DNS еще не разошелся. Для демонстрации можно использовать IP или локальный запуск.

## 10. Мини-чеклист перед защитой

- Проверить, что сайт открывается.
- Проверить логин `admin@example.com` / `admin123`.
- Проверить логин обычного пользователя `test@example.com` / `test123`.
- Открыть заранее страницы `/feed`, `/albums/6`, `/tracks/1`, `/tops`, `/profile`, `/admin`.
- Проверить, что паспорт релиза открывается.
- Проверить, что в админке есть хотя бы одна рецензия для демонстрации статусов.
- Подготовить резервный вариант: локальный запуск через `docker compose up --build`.
- Не показывать на экране пароли сервера, `.env`, токены и приватные ключи.
