## 🎯 Техническое задание

### Цель проекта
Веб-приложение для рецензирования музыкальных альбомов и треков. Пользователи могут просматривать альбомы, оставлять отзывы с детальной системой оценки и взаимодействовать с рецензиями других пользователей.

### Технологический стек

**Frontend:**
- React (JavaScript)
- React Router для маршрутизации
- Axios для HTTP запросов
- Самописные CSS стили с современной тёмно-серой палитрой (темная тема) и "топлёное молоко" (светлая тема)
- Адаптивная верстка
- Компонентный подход

**Backend:**
- Go (Golang) 1.21+
- Gin framework для веб-сервера
- GORM для работы с БД
- PostgreSQL 12+
- Header-based аутентификация (X-User-ID)
- bcrypt для хеширования паролей

### Основной функционал

1. **Главная страница "Актуальное"** - три блока с фоном и скруглёнными углами:
   - Релизы: 5 последних альбомов в одну строку
   - Лайки: 8 популярных треков (2 строки по 4) с карточками
   - Рецензии: 6 популярных рецензий (2 строки по 3)

2. **Страница альбома** - отображение обложки, описания, списка треков и рецензий

3. **Страница трека** - детальная информация о треке, рецензии, ссылка на альбом. Имя артиста жирным и кликабельное

4. **Регистрация и авторизация** - создание пользователя, вход/выход

5. **Профиль пользователя** - просмотр собственных рецензий, редактирование и удаление

6. **Профиль другого пользователя** - просмотр рецензий любого пользователя (переход по нику/аватару из рецензий)

7. **Карточка артиста** - просмотр всех альбомов артиста с статистикой (переход по имени артиста)

8. **Каталог (страница поиска)** - страница с левой панелью фильтров:
   - Поиск по названию/исполнителю
   - Фильтр по жанрам (красивые кнопки, множественный выбор)
   - Сортировка: выпадающий список (по дате выхода, алфавиту, рейтингу, дате добавления)
   - Отображение треков карточками (4 в ряд)
   - Пагинация результатов

9. **Панель администратора** - модерация рецензий, управление контентом

10. **Система лайков** - лайки для альбомов, треков и рецензий

11. **Темная/светлая тема** - переключатель темы с сохранением в localStorage

12. **Поиск** - поиск с автодополнением по названию/исполнителю в Header. Кнопка "Фильтры" открывает страницу каталога

13. **Логотип** - используется в Header рядом с названием сайта

## 🏗 Архитектура проекта

**Слои:**
- `routes/` - определение маршрутов API

- Разделение ответственности между слоями

**Структура:**
- `components/` - переиспользуемые React компоненты
- `pages/` - страницы приложения

- Компонентный подход

- Context API для глобального состояния

## 📊 Модели данных

### User (Пользователь)
- `ID`, `Username`, `Email`, `Password` (хешированный), `IsAdmin`, `CreatedAt`, `UpdatedAt`
- Связи: `Reviews`, `AlbumLikes`, `TrackLikes`, `ReviewLikes`

### Album (Альбом)
- `ID`, `Title`, `Artist`, `GenreID`, `CoverImagePath`, `Description`, `ReleaseDate`, `AverageRating`, `CreatedAt`, `UpdatedAt`
- Связи: `Genre`, `Tracks`, `Reviews`, `Likes`

### Track (Трек)
- `ID`, `AlbumID`, `Title`, `Duration`, `TrackNumber`, `CoverImagePath`, `AverageRating`, `CreatedAt`, `UpdatedAt`
- Связи: `Album`, `Genres` (many-to-many), `Reviews`, `Likes`

### Review (Рецензия)
- `ID`, `UserID`, `AlbumID` (nullable), `TrackID` (nullable), `Text`, `RatingRhymes`, `RatingStructure`, `RatingImplementation`, `RatingIndividuality`, `AtmosphereMultiplier`, `FinalScore`, `Status` (pending/approved/rejected), `ModeratedBy`, `CreatedAt`, `UpdatedAt`
- Связи: `User`, `Album`, `Track`, `Likes`
- **Важно:** Рецензия может быть либо на альбом, либо на трек (не оба одновременно)

### Genre (Жанр)
- `ID`, `Name`, `Description`, `CreatedAt`, `UpdatedAt`
- Связи: `Albums`, `Tracks` (many-to-many)

### Likes (Лайки)
- `AlbumLike`, `TrackLike`, `ReviewLike` - отдельные таблицы с `UserID` и соответствующим ID
- Уникальные индексы на `(user_id, album_id)`, `(user_id, track_id)`, `(user_id, review_id)`

## 🎯 Система оценки

Рецензии оцениваются по специальной системе с 4 базовыми критериями (1-10 баллов) и множителем "Атмосфера/Вайб" (1.0000-1.6072).

### Оценочные параметры

1. **Рифмы / Образы (1–10):**
   Оценка за текст, учитывающая жанровые особенности. Для легкой фоновой поп-музыки или электронной музыки допускаются тривиальные конструкции для максимального балла. Для текстоцентричных жанров требуется сложная рифмовка и глубокий смысл.

2. **Структура / Ритмика (1–10):**
   Включает оценку ритмической составляющей (стихотворный ритм — мелодичность, драматургия частей, контрасты) и гармонию структуры (целостность всех частей трека, концепция альбома и расположение песен).

3. **Реализация стиля (1–10):**
   Оценивает работу исполнителя (качество вокала, речитатив, умение работать с мелодией) и работу звукорежиссера/саунд-продюсера (качество сведения, звучание инструментала).

4. **Индивидуальность / Харизма (1–10):**
   Оценивает уникальность тембра голоса, стиль исполнения, узнаваемость, а также способность артиста передать и погрузить слушателя в эмоции ("верю — не верю" песне).

5. **Атмосфера / Вайб (Множитель) (1-10):**
   Диапазон: от ×1.0000 до ×1.6072 (с точностью до 4 знаков после запятой).
   Субъективная оценка, показывающая, насколько автор сумел передать атмосферу и палитру эмоций композиции.

### Формула подсчёта итогового балла

```
FinalScore = (RatingRhymes + RatingStructure + RatingImplementation + RatingIndividuality) × 1.4 × AtmosphereMultiplier
```

**Пример:** Если пользователь выставил 10 баллов во всех базовых категориях и вайб = 1.6072:
```
(10+10+10+10) × 1.4 × 1.6072 = 40 × 1.4 × 1.6072 = 89.99
```

**Реализация:** Метод `CalculateFinalScore()` в модели `Review` автоматически вычисляет итоговый балл при создании/обновлении рецензии.

## 🎯 API документация

### База URL
База API зависит от режима запуска:

- **Docker Compose dev**: `http://localhost:8080/api` (frontend в браузере ходит в backend на хосте)
- **Docker Compose prod-like**: `/api` (frontend отдаёт nginx и проксирует `/api` в backend внутри compose)
- **Локальный запуск без Docker**: обычно `http://localhost:8080/api`

На фронте это управляется через переменную окружения `REACT_APP_API_URL` (см. раздел “Установка и запуск”).

### Авторизация
Для доступа к защищенным эндпоинтам необходимо передавать заголовок `X-User-ID` с ID пользователя.

### Аутентификация
- `POST /api/auth/register` - Регистрация нового пользователя
- `POST /api/auth/login` - Вход в систему
- `GET /api/auth/me` - Получение информации о текущем пользователе

### Альбомы
- `GET /api/albums` - Получение списка альбомов (с фильтрацией, сортировкой и пагинацией)
  - Параметры: `genre_id`, `search`, `sort_by` (created_at, release_date, title, artist, average_rating), `sort_order` (asc, desc), `page`, `page_size`
- `GET /api/albums/:id` - Получение детальной информации об альбоме
- `GET /api/albums/artist/:name` - Получение всех альбомов артиста по имени (URL-encoded)
- `POST /api/albums` - Создание нового альбома (авторизованный пользователь)
- `PUT /api/albums/:id` - Обновление альбома (только админ)
- `DELETE /api/albums/:id` - Удаление альбома (только админ)
- `POST /api/albums/:id/like` - Лайк альбома
- `DELETE /api/albums/:id/like` - Убрать лайк с альбома

### Рецензии
Рецензии могут быть созданы как для альбомов, так и для треков. При создании необходимо указать либо `album_id`, либо `track_id` (но не оба одновременно).

- `GET /api/reviews` - Получение списка рецензий (с фильтрацией по `album_id`, `track_id`, `user_id`, `status`)
- `GET /api/reviews/popular` - Получение популярных рецензий за последние 24 часа (только с альбомами, сортировка по лайкам)
- `GET /api/reviews/:id` - Получение детальной информации о рецензии
- `POST /api/reviews` - Создание новой рецензии
- `PUT /api/reviews/:id` - Обновление рецензии (автор или админ)
- `DELETE /api/reviews/:id` - Удаление рецензии (автор или админ)
- `POST /api/reviews/:id/like` - Лайк рецензии
- `DELETE /api/reviews/:id/like` - Убрать лайк с рецензии
- `POST /api/reviews/:id/approve` - Одобрение рецензии (только админ)
- `POST /api/reviews/:id/reject` - Отклонение рецензии (только админ)

### Треки
Треки могут иметь несколько жанров (many-to-many связь).

- `GET /api/tracks` - Получение списка треков (с фильтрацией, сортировкой и пагинацией)
  - Параметры: `genre_ids[]` (массив ID жанров, AND логика), `search` (поиск по названию/артисту), `sort_by` (created_at, release_date, title, average_rating, likes_count), `sort_order` (asc, desc), `page`, `page_size`
  - Фильтрация по жанрам: трек должен иметь ВСЕ выбранные жанры (AND логика)
  - Сортировка: по дате добавления, дате выхода альбома, названию, рейтингу, количеству лайков
- `GET /api/tracks/popular` - Получение популярных треков за последние 24 часа
- `GET /api/tracks/:id` - Получение трека по ID
- `GET /api/albums/:id/tracks` - Получение треков альбома
- `POST /api/tracks` - Создание трека (только админ)
- `PUT /api/tracks/:id` - Обновление трека (только админ)
- `DELETE /api/tracks/:id` - Удаление трека (только админ)
- `POST /api/tracks/:id/like` - Лайк трека
- `DELETE /api/tracks/:id/like` - Убрать лайк с трека

### Жанры
- `GET /api/genres` - Получение списка жанров
- `GET /api/genres/:id` - Получение жанра по ID
- `POST /api/genres` - Создание жанра (только админ)
- `PUT /api/genres/:id` - Обновление жанра (только админ)
- `DELETE /api/genres/:id` - Удаление жанра (только админ)

### Поиск
- `GET /api/search?q=query` - Поиск альбомов и треков с автодополнением (возвращает до 5 альбомов и 5 треков). Также возвращает артистов (до 5), которые отображаются первыми в результатах.
  - Артисты в результатах поиска включают поле `cover_image_path` (обложка первого альбома артиста), которая используется как аватар
  - При клике на трек в результатах поиска происходит переход на страницу трека (`/tracks/:id`), а не на страницу альбома

### Пользователи
- `GET /api/users/:id` - Получение пользователя по ID
- `GET /api/users/:id/reviews` - Получение рецензий пользователя (с Preload User, Album, Track, Likes)
- `PUT /api/users/:id` - Обновление профиля пользователя
- `DELETE /api/users/:id` - Удаление пользователя

### Работа с изображениями

Обложки альбомов и треков хранятся в поле `cover_image_path`.

**Формат пути:**
- Путь к изображению указывается относительно корня проекта (например, `/preview/1.jpg`)
- Изображения должны находиться в папке `frontend/public/`
- URL для доступа: `{frontend_origin}/preview/1.jpg` (например, `http://localhost:3000/preview/1.jpg`)

**Обработка ошибок:**
- Если изображение не найдено, отображается placeholder (🎵)
- Все компоненты используют утилиту `getImageUrl()` для формирования правильных URL

## 🚀 Установка и запуск

### Требования
- **Go 1.21+** - [Скачать](https://golang.org/dl/)
- **Node.js 16+** - [Скачать](https://nodejs.org/)
- **PostgreSQL 12+** - [Скачать](https://www.postgresql.org/download/)

Если используете Docker/Compose, достаточно:
- Docker Engine / Docker Desktop (WSL2)
- Docker Compose v2 (`docker compose ...`)

### Быстрая установка

#### Вариант A (рекомендуется): Docker Compose (dev / prod-like)

Из корня репозитория:

```bash
docker compose up --build
```

- Frontend: `http://localhost:3000`
- Backend healthcheck: `http://localhost:8080/healthz`

Prod-like демо:

```bash
docker compose -f compose.prod.yml up --build
```

- Frontend: `http://localhost/`
- API: `http://localhost/api/...` (через nginx proxy `/api`)

Примечание: до внедрения SQL-миграций (Этап 2), при первом запуске prod-like демо может понадобиться временно выставить `MIGRATIONS_MODE=auto`, чтобы схема создалась через AutoMigrate.

#### Вариант B: Локальный запуск без Docker

#### 1. Подготовка базы данных
Убедитесь, что PostgreSQL запущен. База данных создается автоматически при первом запуске backend.

#### 2. Настройка Backend

```bash
cd backend
go mod tidy
```

Создайте файл `.env` в папке `backend/`:
```env
DB_HOST=localhost
DB_PORT=5432
DB_USER=postgres
DB_PASSWORD=postgres
DB_NAME=music_review_db
DB_SSLMODE=disable
PORT=8080
GIN_MODE=debug
```

Запустите сервер:
```bash
go run main.go
```

Backend будет доступен на `http://localhost:8080` (API: `http://localhost:8080/api`)

#### 3. Настройка Frontend

Откройте **новый терминал**:

```bash
cd frontend
npm install
```

Создайте файл `.env` в папке `frontend/`:
```env
REACT_APP_API_URL=http://localhost:8080/api
REACT_APP_USE_MOCK=false
```

Запустите приложение:
```bash
npm start
```

Frontend будет доступен на `http://localhost:3000`

### Тестовые пользователи

После первого запуска автоматически создаются:

- **Администратор:**
  - Email: `admin@example.com`
  - Пароль: `admin123`

- **Тестовый пользователь:**
  - Email: `test@example.com`
  - Пароль: `test123`

### Диагностика базы данных

При запуске backend автоматически выводится детальная диагностика:

**Что логируется:**
- Состояние БД до и после сидинга (количество записей в каждой таблице)
- Процесс создания жанров, пользователей, альбомов, треков, рецензий
- Статистика по созданным/существующим записям
- Предупреждения и ошибки при создании данных

**Таблицы, которые проверяются:**
- `users` - пользователи
- `genres` - жанры
- `albums` - альбомы
- `tracks` - треки
- `track_genres` - связи треков и жанров
- `reviews` - рецензии
- `review_likes`, `track_likes`, `album_likes` - лайки

**Интерпретация логов:**
- `✓ Created` - успешное создание записи
- `ERROR:` - критическая ошибка, требует внимания
- `WARNING:` - предупреждение о возможной проблеме
- `Empty tables detected` - пустые таблицы после сидинга

## 🔧 Технические детали

### Авторизация

**Header-based аутентификация:**
- Используется заголовок `X-User-ID` для идентификации пользователя
- Middleware `auth.go` проверяет наличие заголовка и валидность пользователя
- Middleware `admin.go` проверяет права администратора (`IsAdmin = true`)

**Реализация:**
- Frontend сохраняет `user.id` в `AuthContext` после логина
- Axios interceptor автоматически добавляет `X-User-ID` ко всем запросам
- Backend middleware проверяет заголовок и загружает пользователя из БД

### Работа с базой данных

**Миграции:**
- GORM автоматически создает таблицы при первом запуске
- Все foreign keys настроены автоматически
- Уникальные индексы для лайков предотвращают дубликаты

**Seed данные:**
- Автоматически создаются при первом запуске (если таблицы пустые)
- Создаются: жанры (12 жанров, включая Поп-рок, Инди-поп, Альтернативный рок, Рэп, R&B, Соул), пользователи (админ + тестовые), альбомы, треки, рецензии, лайки
- Треки имеют 2-3 жанра для тестирования множественного выбора жанров
- Используется `FirstOrCreate` для предотвращения дубликатов при перезапуске
- Логирование процесса создания для диагностики

**Диагностика:**
- При запуске выводится состояние БД до и после сидинга
- Проверяется количество записей в каждой таблице
- Выводятся предупреждения при проблемах

### Система лайков

**Реализация:**
- Отдельные таблицы для каждого типа лайков (`album_likes`, `track_likes`, `review_likes`)
- Уникальные индексы на `(user_id, entity_id)` предотвращают дубликаты
- Endpoints для добавления/удаления лайков
- Подсчет количества лайков через GORM associations

**Тестовые данные:**
- Треки получают 5-30 рандомных лайков от разных пользователей (для тестирования "Актуальное")
- Рецензии получают 5-30 рандомных лайков от разных пользователей
- Распределение лайков по времени: 30% в последние 24 часа, 70% за последние 7 дней
- Распределение лайков по разным артистам для разнообразия

### Система модерации рецензий

**Статусы рецензий:**
- `pending` - ожидает модерации (по умолчанию)
- `approved` - одобрена администратором
- `rejected` - отклонена администратором

**Права доступа:**
- Только администратор может одобрять/отклонять рецензии
- Автор рецензии может редактировать/удалять свою рецензию
- Пользователи видят только одобренные рецензии (кроме своих)

### Темная/светлая тема

**Реализация:**
- CSS переменные для цветов темы
- Переключатель темы в Header
- Сохранение выбранной темы в `localStorage`
- Современная тёмно-серая палитра для темной темы (оттенки от #0F0F0F до #1A1A1A)
- Палитра "топлёное молоко" (тёплые кремово-бежевые оттенки) для светлой темы

### Социальные сети в профиле

**Реализация:**
- Поля для VK, Telegram и MAX (ранее Instagram)
- Хранение в JSON формате в поле `social_links` модели User
- Формат: `{"vk": "", "telegram": "", "max": ""}`
- Отображение ссылок в профиле пользователя

### Форма оценки рецензий

**Визуальная структура:**
- Блок базовых критериев (4 оценки): Рифмы, Структура, Реализация, Индивидуальность
- Разделитель с формулой умножения (× 1.4 ×)
- Блок множителя: Атмосфера/Вайб
- Итоговый балл с отображением полной формулы расчета

### Поиск и фильтрация

**Поиск с автодополнением:**
- Endpoint `/api/search?q=query` возвращает до 5 альбомов, 5 треков, 5 артистов
- Артисты отображаются первыми в результатах
- Поиск по названию альбома, названию трека, имени артиста

**Каталог:**
- Фильтрация по жанрам (множественный выбор с AND логикой - трек должен иметь все выбранные жанры)
- Поиск по названию трека или имени артиста
- Сортировка: по дате выхода, алфавиту, рейтингу, дате добавления, количеству лайков
- Пагинация результатов
- Отображение треков карточками (4 в ряд)
- Использует прямой endpoint `/api/tracks` для получения треков с фильтрацией

### Обработка изображений

**Пути к изображениям:**
- Хранятся в БД как `/preview/1.jpg` (относительный путь)
- Физически находятся в `frontend/public/preview/`
- Утилита `getImageUrl()` нормализует путь до вида `/{path}` (браузер резолвит относительно origin, например `{origin}/preview/1.jpg`)

**Обработка ошибок:**
- Если изображение не найдено, отображается placeholder (🎵)
- Все компоненты используют `getImageUrl()` для единообразной обработки

- Проверяйте логи при запуске для диагностики проблем с БД

### Валидация социальных сетей

- **VK**: требует полный URL (type="url")
- **Telegram**: принимает текст, формат @username (type="text")
- **MAX**: принимает текст, поддерживает как URL, так и @username (type="text")

### Аватарки артистов

- На странице артиста (`ArtistPage`) отображается аватар
- Временное решение: используется обложка первого альбома артиста
- Если альбомов нет, отображается placeholder с первой буквой имени
- Размер аватара: 150x150px на десктопе, 120x120px на мобильных

### Загрузка аватарок пользователей

**Backend:**
- Endpoint: `POST /api/users/:id/avatar`
- Принимает multipart/form-data с полем `avatar`
- Валидация: размер до 5MB, форматы JPG, PNG, WEBP
- Сохранение в `frontend/public/avatars/`
- Имя файла: `user_{id}_{timestamp}.{ext}`
- Автоматическое удаление старого аватара при загрузке нового

**Frontend:**
- File input с preview изображения
- Валидация на клиенте (размер, формат)
- Кнопка "Загрузить" для отправки файла
- Fallback на ручной ввод пути/URL (поле `avatar_path` с type="text")
- Индикатор загрузки
- После успешной загрузки вызывается `updateUser` из `AuthContext` для обновления пользователя без логаута
- Превью аватара отображается в круге (border-radius: 50%)

### Система званий пользователей

**Backend:**
- Функция `CalculateUserBadges(userID)` вычисляет звания динамически на основе рецензий
- Анализирует жанры через Album.Genre и Track.Genres
- Звания включены в ответ `GET /api/users/:id` в поле `badges[]`

**Типы званий:**
- **По количеству рецензий:**
  - 🌱 Начинающий критик (1-5)
  - 📝 Опытный критик (6-20)
  - ⭐ Мастер рецензий (21-50)
  - 👑 Легенда критики (51+)
- **По жанрам (5+ рецензий):**
  - 🎷 Джазовый критик
  - 🎤 Поп-эксперт
  - 🎸 Рок-ценитель
  - 🎹 Электронный знаток
  - 🥁 Хип-хоп критик
- **По разнообразию:**
  - 🌈 Универсал (5+ разных жанров)
- **По специализации:**
  - 🎯 Специалист (80%+ рецензий в одном жанре)

**Frontend:**
- Компонент `Badge.js` - отображение одного звания
- Компонент `BadgeList.js` - список званий
- Отображается в `ProfilePage.js` и `UserProfilePage.js` под информацией о пользователе
- Цветовая индикация по приоритету (золото, серебро, бронза)
- Tooltip с описанием при наведении
