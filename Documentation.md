# Техническая документация проекта Mustreview

Mustreview - учебный веб-сервис для музыкальных рецензий и оценок. Пользователь может оценивать альбомы и треки, писать рецензии, лайкать материалы, подписываться на авторов и вести музыкальный профиль с предпочтениями, статистикой, уровнем и достижениями.

Документ актуален для текущего состояния проекта и заменяет разрозненные заметки из `docs/` и `Roadmap.md`.

## 1. Стек

| База данных | PostgreSQL |
| Инфраструктура | Docker Compose, Nginx для prod-like запуска |

## 2. Быстрый запуск

```bash
docker compose up --build
```

После запуска:

| Сервис | URL |
| --- | --- |
| Frontend | `http://localhost:3000` |
| Backend API | `http://localhost:8080/api` |
| PostgreSQL | `localhost:5432` |

Тестовые пользователи создаются сидером:

| Роль | Email | Пароль |
| --- | --- | --- |
| Администратор | `admin@example.com` | `admin123` |
| Пользователь | `test@example.com` | `test123` |

Полезные команды проверки:



## 6. Модели данных

### User

Ключевые поля:

| Поле | Описание |
| --- | --- |
| `username`, `email`, `password` | учетные данные, пароль хранится как bcrypt hash |
| `is_admin` | доступ к админке |
| `bio`, `avatar_path`, `social_links` | оформление профиля |
| `favorite_album_ids` | JSON-массив ID любимых альбомов |
| `favorite_track_ids` | JSON-массив ID любимых треков |
| `favorite_artists` | JSON-массив выбранных артистов |
| `is_verified_artist` | отметка верифицированного артиста |

### Album

Альбом содержит название, артиста, жанр, описание, обложку и агрегированную среднюю оценку. Связан с треками, рецензиями и лайками.

### Track

Трек привязан к альбому, имеет номер, длительность, жанры, обложку и среднюю оценку.

### Review

Рецензия относится либо к альбому, либо к треку. Содержит текст, пять параметров оценки и итоговый балл. Статус модерации: `pending`, `approved`, `rejected`.

### Likes

Лайки разделены по сущностям: альбомы, треки и рецензии. Для каждой пары `user_id + entity_id` действует уникальность.

### UserFollow

Связь подписки: `follower_id` подписан на `following_id`.

## 7. API

Базовый URL: `http://localhost:8080/api`.

Авторизация использует подписанный bearer-token, который возвращается после входа или регистрации и передается в заголовке `Authorization: Bearer ...`. Для локальной разработки сохранен fallback `X-User-ID`, но в production compose он отключен через `AUTH_ALLOW_USER_ID_HEADER=false`.

Сессионные параметры:

| Переменная | Описание |
| --- | --- |
| `SESSION_SECRET` | секрет подписи токена |
| `SESSION_TTL_HOURS` | срок жизни токена, по умолчанию 168 часов |
| `AUTH_ALLOW_USER_ID_HEADER` | разрешает dev-fallback через `X-User-ID` |

### Auth

| Метод | Путь | Описание |
| --- | --- | --- |
| `POST` | `/auth/register` | регистрация |
| `POST` | `/auth/login` | вход |
| `GET` | `/auth/me` | текущий пользователь |

### Albums и tracks

| Метод | Путь | Описание |
| --- | --- | --- |
| `GET` | `/albums` | список альбомов с фильтрами |
| `GET` | `/albums/:id` | альбом по ID |
| `GET` | `/albums/:id/tracks` | треки альбома |
| `GET` | `/tracks` | список треков с фильтрами |
| `GET` | `/tracks/:id` | трек по ID |
| `POST/DELETE` | `/albums/:id/like`, `/tracks/:id/like` | лайк/снятие лайка |

### Reviews

| Метод | Путь | Описание |
| --- | --- | --- |
| `GET` | `/reviews` | список рецензий, поддерживает фильтры |
| `GET` | `/reviews/:id` | рецензия по ID |
| `POST` | `/reviews` | создать рецензию |
| `PUT` | `/reviews/:id` | обновить рецензию |
| `DELETE` | `/reviews/:id` | удалить рецензию |
| `POST/DELETE` | `/reviews/:id/like` | лайк/снятие лайка |
| `POST` | `/reviews/:id/approve` | одобрить, только admin |
| `POST` | `/reviews/:id/reject` | отклонить, только admin |

### Users

| Метод | Путь | Описание |
| --- | --- | --- |
| `GET` | `/users/:id` | пользователь, статистика, предпочтения, подписки |
| `GET` | `/users/:id/reviews` | рецензии пользователя |
| `GET` | `/users/:id/liked-reviews` | рецензии, которые пользователь лайкнул |
| `PUT` | `/users/:id` | обновить профиль |
| `POST` | `/users/:id/avatar` | загрузить аватар |
| `PUT` | `/users/:id/favorites` | сохранить предпочтения |
| `POST/DELETE` | `/users/:id/follow` | подписка/отписка |

`PUT /users/:id/favorites` принимает:

```json
{
  "artist_names": ["Баста"],
  "album_ids": [1, 2, 3],
  "track_ids": [4, 5, 6]
}
```

Каждая категория ограничена тремя элементами.

## 8. Система оценки

Пользователь оценивает релиз по четырем основным параметрам и атмосфере:

| Параметр | Диапазон |
| --- | --- |
| Рифмы / образы | 1-10 |
| Структура / ритмика | 1-10 |
| Реализация стиля | 1-10 |
| Индивидуальность / харизма | 1-10 |
| Атмосфера / вайб | 1-10, внутри переводится в множитель |

Итоговая оценка приводится примерно к шкале 1-90. В интерфейсе формула скрыта от пользователя: показывается крупный итог, ниже маленькие числа, а в подсказке - понятное объяснение "из чего складывается оценка" без технических коэффициентов.

Для альбомов и треков средняя оценка показывается целым числом. В подсказке также используются округленные значения, чтобы интерфейс не выглядел перегруженным.

## 9. Профиль пользователя

Профиль вдохновлен подходом РЗТ, но адаптирован под Mustreview:

- слева узкая колонка с аватаром, данными, уровнем и статистикой;
- справа крупная музыкальная карточка и вкладки;
- вкладка "Предпочтения" показывает артистов, альбомы и треки;
- вкладка "Рецензии и оценки" показывает материалы пользователя;
- вкладка "Понравилось" показывает лайкнутые рецензии;
- вкладка "Достижения" показывает бейджи.

### Предпочтения

Пользователь может вручную выбрать до трех артистов, альбомов и треков. Если ручной выбор пустой, профиль автоматически собирает блоки из рецензий пользователя, чтобы страница не была пустой.

В редакторе предпочтений стартовое состояние совпадает с тем, что видно на профиле: если треки были собраны автоматически, они тоже появляются в слотах редактирования и могут быть сохранены как личный выбор.

### Уровень профиля

Уровень считается не по средней оценке, а по активности и реакции сообщества. Это важно: пользователь не должен получать больше опыта только потому, что ставит релизам более высокие оценки.

Формула опыта:

| Действие | Опыт |
| --- | --- |
| Рецензия | +320 |
| Лайк к рецензии пользователя | +55 |
| Лайк, поставленный пользователем | +12 |
| Авторский лайк | +240 |

Пороговые уровни:

| Уровень | Баллы |
| --- | --- |
| Бронзовый | 0 |
| Серебряный | 2 500 |
| Золотой | 8 000 |
| Изумрудный | 18 000 |
| Платиновый | 36 000 |

Рядом с уровнем есть подсказка `i`, где объясняется начисление опыта.

## 10. Дизайн и темы

В проекте есть светлая и темная темы через CSS variables. Главные правила текущего дизайна:

- светлая тема теплая, но не должна сливаться в один бежевый слой;
- карточки должны отличаться от фона за счет контраста, границ и теней;
- кнопки и сегменты используют единый pill-стиль со скользящим индикатором;
- оценки показываются компактным стильным блоком без сокращений на самом бейдже;
- tooltip не должен проваливаться под соседние элементы;
- ПК-версия в приоритете, мобильная должна оставаться аккуратной и без горизонтального развала.
- системные `alert` / `confirm` не используются: действия должны происходить внутри интерфейса проекта.

## 11. DevOps и CI/CD

DevOps-часть вынесена как отдельная сильная сторона проекта:

- локальный `docker-compose.yml` поднимает PostgreSQL, backend и frontend в dev-режиме;
- `compose.prod.yml` собирает production-like стек с nginx frontend и backend healthcheck;
- `compose.deploy.yml` запускает готовые backend/frontend образы из реестра, что подходит для VPS;
- GitHub Actions выполняет `go vet`, `go test`, backend build и frontend build;
- pipeline валидирует compose-файлы, поднимает production stack и проверяет HTTP health endpoints;
- после успешных проверок CI собирает и публикует Docker-образы backend/frontend в GHCR.

## 12. Демо-данные

Демо-данные создаются в `backend/database/database.go`. Сидер работает идемпотентно: при повторном запуске он не дублирует уже созданные сущности, но досоздает недостающие данные для демонстрации.

Сейчас сидер наполняет:

- базовые аккаунты `admin@example.com` / `test@example.com`;
- набор демо-пользователей с разными никами и описаниями;
- верифицированные аккаунты артистов для сценариев авторских реакций;
- альбомы, треки, жанры и связи треков с жанрами;
- одобренные рецензии для ленты и профилей;
- pending-рецензии для панели модерации;
- лайки альбомов, треков и рецензий.

Отдельной сущности комментариев в текущей модели нет: роль пользовательской активности сейчас выполняют рецензии, лайки и подписки.

## 13. Изображения и аватары

Обложки лежат в `frontend/public/preview`. Аватары пользователей загружаются в `frontend/public/avatars`.

В `docker-compose.yml` добавлен volume для аватаров:

```yaml
./frontend/public/avatars:/frontend/public/avatars
```

Это нужно, чтобы загруженные изображения не терялись при перезапуске контейнера.

## 15. Ближайшие задачи

1. Довести визуальный стиль светлой темы: добавить больше разделения слоев, меньше одинаковых оттенков.
2. Продолжить улучшение ленты: фильтры по жанрам, подборки, компактные рекомендации или "лучшее за день".
3. Решить вопрос с бесконечной лентой: делать только если будет понятная пагинация, skeleton-состояния и сохранение позиции при переходах.
4. Доработать достижения: прогресс до следующего достижения и больше сценариев получения.
5. Продумать авторские лайки: кто может ставить, как хранить, как показывать в профиле.
6. Улучшить мобильную версию после стабилизации ПК-дизайна.
7. Для дипломной записки подготовить отдельный `.docx` по шаблону вуза, используя этот документ как техническую основу.

## 16. Важные ограничения учебного проекта

- Dev-fallback через `X-User-ID` оставлен только для локальной разработки и отключается в production.
- Сидер используется для демо-данных и может перетирать ожидания при пересоздании БД.
- Часть данных профиля собирается автоматически, если пользователь не заполнил личные предпочтения.
- Документация намеренно хранится в одном файле, чтобы не расползалась перед подготовкой дипломной записки.
