# План подготовки ВКР Афонькина М.А.

## Базовые требования оформления

- Формат страницы: А4.
- Шрифт основного текста: Times New Roman, кегль 14.
- Межстрочный интервал: 1,5.
- Поля по СМК: левое 30 мм, правое 10 мм, верхнее 20 мм, нижнее 20 мм.
- Абзацный отступ: 1,25 см.
- Нумерация страниц: внизу по центру, титульный лист и задание входят в общую нумерацию, но номера на них не ставятся.
- Разделы основной части начинаются с новой страницы.
- Таблицы и рисунки должны иметь сквозную нумерацию и ссылки в тексте.
- Листинги программного кода выносятся в приложения.

Техническая политика текста: не использовать скрытые символы, искусственные замены букв, невидимые вставки и другие приемы, которые портят документ. Текст пишется оригинально, с нормальными ссылками на источники и без копирования чужих фрагментов.

## Рабочая тема

Разработка информационной системы рецензирования музыкального творчества.

Тему можно уточнить ближе к финальной версии:

- "Разработка веб-приложения для рецензирования и рейтинговой оценки музыкального творчества";
- "Проектирование и разработка информационной системы рецензирования музыкального творчества";
- "Разработка веб-сервиса для коллективного рецензирования современной музыки".

## Планируемый объем

Цель: 60+ страниц без учета приложений или около 60-75 страниц вместе с приложениями, если кафедра допускает учет приложений в общем объеме. Для надежности основную часть лучше довести до 50-60 страниц, а листинги и CI/CD добавить в приложения.

Ориентировочный бюджет:

| Раздел | Объем |
| --- | ---: |
| Титульный лист, задание, реферат, содержание | 4-6 стр. |
| Введение | 3-4 стр. |
| Глава 1 | 15-18 стр. |
| Глава 2 | 22-28 стр. |
| Глава 3 | 10-14 стр. |
| Заключение и источники | 4-6 стр. |
| Приложения | 12-25 стр. |

## Предлагаемая структура

### РЕФЕРАТ

Кратко описать объект, цель, методы, результат, стек, практическую значимость. Ключевые слова: ИНФОРМАЦИОННАЯ СИСТЕМА, ВЕБ-ПРИЛОЖЕНИЕ, РЕЦЕНЗИРОВАНИЕ, МУЗЫКАЛЬНЫЙ КОНТЕНТ, МНОГОКРИТЕРИАЛЬНАЯ ОЦЕНКА, МОДЕРАЦИЯ, DOCKER, CI/CD.

### ВВЕДЕНИЕ

Актуальность, объект, предмет, цель, задачи, практическая значимость. Основу взять из курсовой, но заменить формулировки "курсовой проект" на "выпускная квалификационная работа" и расширить фокус до готовой системы.

### 1 Теоретические основы и анализ существующих решений рецензирования музыкального творчества

1.1 Характеристика предметной области и областей применения онлайн-рецензирования русскоязычной современной музыки.

1.2 Анализ исходных данных и информационных потоков в процессе рецензирования музыкального творчества на веб-платформах.

1.3 Оценка существующих веб-платформ и практических решений для рецензирования музыкальных произведений.

1.4 Цель и задачи разработки информационной системы рецензирования музыкального творчества.

1.5 Выводы по первой главе.

### 2 Проектирование и разработка информационной системы рецензирования музыкального творчества

2.1 Архитектура веб-приложения.

Описать трехзвенную архитектуру: клиентское приложение React, серверное API Go/Gin, PostgreSQL. Отдельно отметить Docker Compose как средство воспроизводимого запуска.

Рисунок: "Архитектура информационной системы Mustreview".

2.2 Функциональные требования и сценарии использования.

Описать роли: гость, пользователь, администратор/модератор, верифицированный аккаунт артиста. Сценарии: просмотр ленты, поиск релиза, написание рецензии, оценивание, лайк, настройка профиля, модерация, добавление альбома и треков.

Таблица: "Функциональные требования к системе".

2.3 Проектирование базы данных.

Описать сущности: users, albums, tracks, genres, reviews, likes, follows, achievements/favorites. Пояснить связи, JSON-поля избранного, хранение пароля как bcrypt hash.

Рисунок: "ER-диаграмма базы данных".

2.4 Модель рейтинговой оценки и геймификации.

Описать критерии: рифмы/образы, структура/ритмика, реализация стиля, индивидуальность/харизма, атмосфера/вайб. Пояснить итоговую оценку без раскрытия пользовательских внутренних коэффициентов как "нормализация оценки". Описать уровни профиля и достижения.

Таблица: "Критерии экспертной оценки рецензии".

2.5 Разработка серверной части.

Описать REST API, middleware авторизации, сессии, загрузку файлов, сидирование демо-данных, модерацию рецензий, создание релизов администратором.

Рисунок: "Схема обработки запроса создания рецензии".

2.6 Разработка клиентской части.

Описать React-компоненты: лента, карточки рейтингов, профиль, редактируемые предпочтения, глоссарий геймификации, панель администратора, светлая/темная темы.

Рисунок: "Структура пользовательского интерфейса".

2.7 Разработка административной панели.

Описать модерацию рецензий, переключатель статусов, добавление альбомов и треков, загрузку обложек, генерацию длительности треков с возможностью ручной корректировки.

2.8 DevOps-инструменты и воспроизводимость развертывания.

Сделать сильный раздел диплома:

- Dockerfile для backend и frontend;
- docker-compose.yml для разработки;
- compose.prod.yml для production-like запуска;
- healthcheck backend/frontend/db;
- GitHub Actions pipeline: go vet, go test, backend build, frontend build, compose smoke test, сборка Docker-образов;
- публикация образов в GHCR;
- переменные окружения и разделение dev/prod режимов;
- преимущества для сопровождения и демонстрации проекта.

Рисунок: "CI/CD-процесс проекта".

Таблица: "Этапы pipeline GitHub Actions".

2.9 Выводы по второй главе.

### 3 Результаты опытной эксплуатации и проверка работоспособности системы

3.1 Подготовка демонстрационных данных.

Описать сидеры: пользователи, тестовые аккаунты, аккаунты артистов, альбомы, треки, рецензии, лайки, подписки.

3.2 Пользовательский сценарий работы с системой.

Сценарии от входа до публикации рецензии и отображения рейтинга.

3.3 Сценарий работы администратора.

Модерация рецензии, добавление релиза, загрузка обложки, проверка результата на странице альбома.

3.4 Проверка качества интерфейса.

Описать светлую/темную темы, адаптивность, читаемость карточек, подсказки рейтинга, отсутствие системных alert-окон.

3.5 Проверка сборки и развертывания.

Показать результаты команд: npm build, go test, go vet, docker compose smoke. Можно вставить фрагменты логов в приложение.

3.6 Анализ результатов и ограничения.

Итоги, что реализовано, что можно развивать: бесконечная лента, рекомендации, расширенная аналитика, более строгий RBAC, полноценное объектное хранилище для файлов.

3.7 Выводы по третьей главе.

### ЗАКЛЮЧЕНИЕ

Сопоставить цель и задачи с результатом. Отдельно отметить практическую значимость: готовый веб-сервис, демо-данные, модерация, геймификация, CI/CD и контейнеризация.

Ориентир: 25-35 источников. Нужны источники по веб-разработке, React, Go, PostgreSQL, Docker, CI/CD, UX, музыкальным платформам и нормативным документам.

### Приложения

- Приложение А: структура API.
- Приложение Б: листинг моделей базы данных.
- Приложение В: листинг контроллера рецензий или рейтингового блока.

- Приложение Д: конфигурация GitHub Actions.
